Перенос информационной системы на платформу «Гиперион»¶

Существующую информационную систему можно перенести на платформу «Гиперион» и в дальнейшем управлять этой системой с помощью инструментов Платформы. Перенос системы включает пять этапов:

  1. Проверка соответствия системы требованиям и подготовка системы.
  2. Подготовка Платформы.
  3. Сборка сервисов.
  4. Добавление стендов.
  5. Формирование релизов и установка на стенды.

Проверка и подготовка системы¶

Для размещения на Платформе система должна соответствовать пяти основным требованиям:

  • Микросервисная архитектура системы. Платформа предназначена для систем на базе микросервисов. При размещении системы с иной архитектурой необходимо проконсультироваться с разработчиками Платформы.
  • Наличие каталогов LDAP. Для составления списка пользователей Платформы используются внешние (относительно Платформы) LDAP-каталоги.
  • Организация репозиториев Git. Для каждого сервиса или библиотеки в составе системы должен быть создан отдельный репозиторий.
  • Сборка сервисов под архитектуру x86. Платформа не поддерживает другие процессорные архитектуры.
  • Использование допустимых шаблонов CI/CD.

Допустимые шаблоны CI/CD¶

Платформа поддерживает следующие шаблоны CI/CD, необходимые для автоматизации процессов сборки и публикации артефактов:

  • Net Core 3.1,
  • .Net 5.0,
  • .Net 6.0,
  • Node.js v14.0.0,
  • Node.js v16.0.0,
  • Node.js v18.0.0,
  • Node.js v19.0.0,
  • Java 8, Maven,
  • Java 8, Gradle,
  • Java 11, Maven,
  • Java 11, Gradle,
  • Java 17, Maven,
  • Java 17, Gradle,
  • Nginx со статическими файлами.

Эти шаблоны оптимизированы для сборки и использования в Платформе, так как собираются на основе легковесных базовых образов и заранее подготовлены для работы в среде с ограниченными правами пользователя.

Подготовка Платформы¶

Подготовка Платформы к переносу системы включает:

  • Добавление серверной площадки,
  • Добавление информационной системы,
  • Добавление или импорт сервисов в составе системы.

Добавление серверной площадки¶

Серверная площадка — это доступный платформе «Гиперион» кластер Kubernetes, в котором размещаются стенды информационных систем. Платформа поддерживает подключение серверных площадок на базе следующих дистрибутивов Kubernetes:

  • OCP,
  • OKD,
  • Kubernetes,
  • Rancher.

Чтобы подключить серверную площадку к Платформе, необходимо установить выбранный дистрибутив на сервера и получить следующие данные:

  • URL API кластера Kubernetes вида https://api.cluster.example.com:6443,
  • Bearer Token сервисного аккаунта с уровнем доступа cluster-admin,
  • Console URL вида https://console.apps.cluster.example.com.

Когда дистрибутив Kubernetes установлен и данные получены, нужно добавить серверную площадку на Портале разработчика с помощью формы добавления серверной площадки.

Добавление информационной системы¶

Система добавляется на Портале разработчика с помощью формы добавления информационной системы.

Когда система добавлена, в компоненте GitLab автоматически создается группа и инфраструктурный репозиторий hyperion. По мере включения в систему сервисов в этот репозиторий добавляются конфигурации kustomize для шаблонизации сервисов и стендов системы.

Добавление и импорт сервисов¶

Сервис — это приложение в составе информационной системы, ее минимальный функциональный компонент. Включение сервисов в состав системы производится на Портале разработчика с помощью форм добавления и импорта.

В корне репозитория сервиса Платформа автоматически создает файл hyperion-ci.yml, использующий шаблоны Платформы для обеспечения процесса CI/CD.

Для импорта репозиториев необходима учетная запись с правами Reporter на сервере GitLab, с которого будет производиться импорт, а также наличие сетевой связанности до него от сервисов Платформы.

Репозитории в импортированной группе должны быть организованы без подгрупп.

При импорте или создании сервиса Платформа позволяет сгенерировать набор конфигурации kustomize для развертывания приложения. После создания такой набор автоматически добавляется в инфраструктурный репозиторий. Эту конфигурацию впоследствии можно перенастроить вручную для реализации инфраструктурных особенностей сервиса. Конфигурация может быть сгенерирована для следующих объектов Kubernetes:

  • Deployment,
  • Service,
  • Route,
  • ServiceMonitor.

При переносе на Платформу существующего набора конфигурации рекомендуется проконсультироваться с разработчиками Платформы.

Сборка сервисов¶

Сборка — это результат перевода исходного программного кода сервиса в исполняемый код (артефакт). Сборка сервисов производится на Портале разработчика с помощью формы добавления сборки. Для контроля версий сборок рекомендуется использовать семантическое версионирование.

При создании сборки сервиса в соответствующем репозитории создается тег и запускается конвейер из шаблона CI/CD. Если версия сборки, создаваемой на Портале разработчика, соответствует версии тега, который был импортирован вместе с репозиторием, необходимо проконсультироваться с разработчиком Платформы.

Зависимости для образов и сборок должны быть размещены в хранилище артефактов Платформы.

Кастомизация Dockerfile или сборочного процесса реализуется через переменные окружения и переопределение сборки в конфигурационном файле hyperion-ci.yml. Рекомендуется использовать для сборки образов контейнеров сертифицированные и легковесные базовые образы. Это обеспечивает максимальное быстродействие цикла CI/CD и устраняет большую часть проблем информационной безопасности. Оптимальными для использования являются образы, которые собираются шаблонами Платформы.

При сборке образа сервиса происходит его анализ на уязвимости для идентификации возможных CVE.

Совет: добавляйте юнит-тесты при разработке сервисов

При разработке сервисов рекомендуется добавлять юнит-тесты. Это позволяет пользователям Платформы, в частности операторам систем, анализировать результат прохождения тестов на Портале разработчика и убеждаться в отсутствии регрессионных проблем. Тесты рекомендуется разрабатывать с учетом распространенных в языках программирования лучших практик и подходов.

Также рекомендуется уделять внимание прохождению статического анализа кода в компоненте Платформы SonarQube, чтобы отслеживать ошибки программирования, проблемы с безопасностью и отклонения от стандартных практик написания кода.

Добавление стендов¶

Стенд — это набор ресурсов серверной площадки, предназначенный для установки релизов. Добавление стенда производится на Портале разработчика с помощью формы добавления стенда. При подготовке стенда Платформа автоматически создает на целевой серверной площадке следующие объекты Kubernetes:

  • Namespace, в котором будет размещена сборка,
  • Quota — ограничение запрашиваемых ресурсов CPU, Memory и Storage,
  • LimitRanges — базовые значения ресурсов для контейнеров без установленных ограничений Request/Limits,
  • RBAC-объекты для авторизации пользователей Платформы.

В инфраструктурном репозитории hyperion создается директория нового стенда в overlays. Это позволяет отредактировать конфигурацию системы для конкретного стенда.

Формирование релизов и установка на стенды¶

Чтобы установить сборку сервиса на стенд, необходимо сформировать релиз.

Релиз — это выпуск новой версии одного или нескольких сервисов, входящих в информационную систему. Релиз формируется на Портале разработчика с помощью формы добавления релиза. Для установки необходимо выбрать действие «Установить на стенд» в целевой строке списка релизов или на карточке релиза.

Note

Возможность установки зависит от текущего статуса релиза. Установить релиз на стенд для разработки можно в любом статусе, на продуктовый стенд — только в статусах «Согласован» и «Установлен».

При установке релиза на стенд Платформа производит коммит в kustomization.yaml, расположенный в соответствующей стенду директории внутри overlays инфраструктурного репозитория hyperion. Коммит содержит перечисленные в релизе версии сервисов.
